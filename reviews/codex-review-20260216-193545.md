[CRITICAL] gateway/tests/test_main_endpoints.py:33–53 — Integration tests invoke the real forward-to-agent pipeline without mocking HTTP calls  
The new `test_main_endpoints.py` spins up the FastAPI app and exercises `/forward`, but never stubs or mocks out `MultiAgentRouter.forward_to_agent`. As a result, every “forward” test will attempt a real HTTP request to the default OpenClaw container (`http://openclaw:18789`), causing test timeouts or connection errors on CI.  
Suggested fix: in your `client` fixture, patch `MultiAgentRouter.forward_to_agent` (or the underlying `httpx` client) to return a dummy success response. For example:  
```python
with patch('gateway.ingest_api.router.MultiAgentRouter.forward_to_agent', new_callable=AsyncMock) as fake_forward:
    fake_forward.return_value = ("OK", 200)
    async with AsyncClient(...):
        yield ac
```  
This isolates the gateway logic from external network calls and ensures your integration tests remain fast and reliable.

[MEDIUM] gateway/ingest_api/main.py:142–150 — Hard-coded CORS origins rather than config-driven  
You’ve guarded the chat UI with `CORSMiddleware` and hard-coded origins (`localhost:8080`, etc.). This means any environment change (different port/host) requires code edits. It also risks allowing unauthorized origins if someone forgets to update.  
Suggested fix: expose a `GatewayConfig.cors_origins: list[str]` field, load it from your YAML or `.env`, and pass it to `add_middleware` so origins are declared in config, not code.
[MEDIUM] gateway/ingest_api/config.py:40 — No validation on `default_url` allows SSRF or misconfiguration  
The new `default_url` field in `RouterConfig` is a free-form `str`. If an administrator accidentally or maliciously points it to `http://169.254.169.254/`, agents could be tricked into leaking cloud-metadata.  
Suggested fix: use Pydantic’s `HttpUrl` type instead of `str`, and optionally validate the host portion against an allow-list (`openclaw`, `localhost`, etc.).
[MEDIUM] gateway/ingest_api/sanitizer.py:325–332 — Complex regex in `block_credentials` may hurt performance on large messages  
The replacement for TOTP and high-entropy strings now uses multiple lookaheads and longer patterns. Since `block_credentials()` runs on every forwarded response, these regexes may be expensive on large payloads.  
Suggested fix: precompile these patterns once during the `PIISanitizer` initialization (`re.compile(...)`) rather than re-compiling on each call; consider bounding the input size or scanning incrementally.

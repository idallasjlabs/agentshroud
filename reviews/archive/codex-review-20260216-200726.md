[HIGH] gateway/ingest_api/main.py:144–151 — Hard-coded CORS origins  
  The `CORSMiddleware` is wired up with a fixed list of `allow_origins` (`localhost:8080`, `127.0.0.1:8080`, `localhost:18790`, `127.0.0.1:18790`) rather than using the new `GatewayConfig.cors_origins`. Any UI port or hostname change will require a code change, and there is risk of unintentionally exposing a permissive origin.  
  Suggested fix: Read `cors_origins` from your loaded configuration and pass that list to `add_middleware`, e.g.  
  ```python
  origins = app_state.config.cors_origins
  app.add_middleware(CORSMiddleware, allow_origins=origins, …)
  ```

[MEDIUM] gateway/ingest_api/config.py: URL validation could be strengthened  
  The custom `field_validator` for `default_url` and `targets` enforces scheme and restricts hostnames to localhost/OpenClaw, but uses simple `startswith` checks and a manual host allow-list. It does not validate port ranges or catch malformed URLs (e.g. missing netloc).  
  Suggested fix: Use Pydantic’s `HttpUrl` or `AnyHttpUrl` type with a custom `validate_allowed_host` to enforce the same host allow-list and port constraints, enabling full parsing and error messages.  
[MEDIUM] gateway/ingest_api/sanitizer.py: regex patterns may false-positive and degrade performance  
  The new pre-compiled TOTP (`self._totp_pattern`) and entropy (`self._entropy_pattern`) patterns use look-aheads and broad character classes. On large messages, these can be expensive and may still block legitimate content (e.g. “Order #123456 shipped”).  
  Suggested fix:  
  - Tune the patterns to require clear contextual keywords (e.g. “TOTP code:”) rather than any six-digit run.  
  - Limit input chunk size or scan in streaming fashion.  
  - Benchmark on representative payloads and adjust as needed.

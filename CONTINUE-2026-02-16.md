## 🔴 Known Limitations

### 1. Gmail Channel Not Supported
- **Issue**: OpenClaw does not support Gmail as an inbound channel
- **Available Channels**: telegram, whatsapp, discord, slack, signal, imessage, etc.
- **Workaround**: Bot can SEND emails via AI tools, but cannot RECEIVE emails
- **Status**: Limitation of OpenClaw platform, not SecureClaw

### 2. Browser-Fetch Not Auto-Discovered
- **Issue**: Bot doesn't automatically use browser-fetch skill when sent 1Password links via Telegram
- **Current Behavior**: Bot tries `web_fetch` (fails) or native `browser` tool (not available)
- **Workaround**: Manual invocation via Docker exec (works perfectly)
- **Future Fix**: Publish skill to ClawHub or configure custom skill loading
- **Status**: Skill is fully functional, just needs registration/publication

### 3. Seccomp Profiles Disabled
- **Issue**: Seccomp profiles exist but are commented out in docker-compose.yml
- **Reason**: Caused thread creation issues in previous testing
- **Risk Level**: MEDIUM (other security layers still active)
- **Next Steps**: Profile syscalls with strace, add missing syscalls, re-enable
- **File**: `docker/docker-compose.yml` lines 29, 104-105

### 4. OpenClaw Container Not Read-Only
- **Issue**: `read_only: false` in docker-compose.yml for openclaw service
- **Reason**: Needs write access to config/workspace volumes
- **Risk Level**: MEDIUM (container is isolated, no host access)
- **Next Steps**: Change to `read_only: true`, add tmpfs for write paths
- **File**: `docker/docker-compose.yml` line 113

## 🎓 What We Learned This Session

### Technical Insights
1. **OpenClaw device pairing is multi-layered**: Gateway password + per-device approval
2. **1Password share links require JavaScript**: Simple HTTP fetch won't work
3. **Playwright in Docker needs persistent storage**: Browser binaries are large (929MB)
4. **Docker tmpfs with noexec breaks browsers**: Need to remove noexec flag or use volume
5. **Custom skills need registration**: Files in skills directory aren't auto-discovered
6. **Safari Private Windows are separate devices**: Each session requires new pairing

### Security Lessons
1. **Never compromise security for convenience**: Device pairing is critical, keep it enabled
2. **Dual-layer auth is powerful**: Password + device pairing prevents many attacks
3. **Audit everything**: Logs are essential for security and debugging
4. **Isolation works**: Container has no host access, no LAN access - exactly as designed
5. **Plain text is the enemy**: 1Password share links avoid sending credentials in Telegram

### Workflow Improvements
1. **Documentation is critical**: Device pairing doc saved hours of confusion
2. **Test incrementally**: Verify each component works before integration
3. **Manual workarounds enable progress**: Browser-fetch works now, automation can wait
4. **User control is key**: Approval on every sensitive action (SecureClaw principle)

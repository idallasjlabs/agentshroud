## Key Decisions & Context

### 1. Complete Architectural Pivot
**Decision**: Abandon "One Claw Tied Behind Your Back" (hardened container only) in favor of SecureClaw (proxy layer framework)

**Rationale**:
- Container-only approach still requires OAuth to real accounts → standing access vulnerability
- Compromised OpenClaw with OAuth tokens = complete breach
- Proxy layer eliminates standing access entirely by requiring user-initiated forwarding
- User controls data flow, not agent

### 2. Archive vs. Delete Old Work
**Decision**: Archive old documentation to `docs/archive/`, delete old implementation files

**Rationale**:
- Old docs contain valuable security analysis and session history (101KB)
- Implementation files (scripts, containers, configs) no longer needed
- Clean break allows fresh start without legacy code confusion

### 3. iOS Shortcuts vs. Custom App
**Decision**: Use native iOS Shortcuts for data forwarding, not custom app

**Rationale**:
- Zero app store approval process
- Native Share Sheet integration
- On-device processing (voice, OCR, EXIF stripping)
- No third-party dependencies
- Users already familiar with Shortcuts API

### 4. SQLite vs. PostgreSQL for Data Ledger
**Decision**: SQLite for gateway data ledger

**Rationale**:
- Local-only (no cloud database)
- Single file (easy backup)
- No separate service to manage
- Sufficient performance for single-user workload

### 5. Configuration-as-Code
**Decision**: Create `secureclaw.yaml` with all settings, not environment variables only

**Rationale**:
- Version-controlled security posture
- Drift detection possible (compare running config to yaml)
- Easier to audit and review
- Single source of truth for all components

### 6. MIT License
**Decision**: MIT License for SecureClaw

**Rationale**:
- Most permissive open source license
- Encourages adoption and contribution
- Compatible with OpenClaw's existing license
- No restrictions on commercial use

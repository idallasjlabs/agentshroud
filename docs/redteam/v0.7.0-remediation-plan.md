## Critical Finding

> **Effective enforcement rate across all 33 modules × 4 loss categories = 0%.**
>
> Thirty-two modules default to monitor mode. The single module claiming enforcement (API Key Vault #24) is contradicted by evidence.

## Tier 1: Must-Have (Deployment Blockers) — v0.7.0 Scope

### Sprint 1: Enforce-by-Default (R-02, R-03)
**Weighted Score: 4.8 (highest priority)**
**Remediation: `01-enforce-by-default.md`**
**Status: ⚠️ Partially done in v0.6.0 (FileSandbox only)**

- [ ] PII Sanitizer (#1) → enforce mode (redact, not log-only)
- [ ] Prompt Injection Defense (#8) → enforce mode (block, not log-only)
- [ ] Egress Filtering (#10) → enforce mode with domain allowlist
- [ ] MCP Proxy (#19) → enforce mode with per-tool permissions
- [ ] Monitor mode requires explicit opt-in (`--permissive` flag or `AGENTSHROUD_MODE=monitor`)
- [ ] Document risk of monitor mode

**Verification:**
- Send `4111 1111 1111 1111` → must be redacted, not echoed
- Send prompt injection → must be blocked
- Request egress to unknown domain → must be denied
- Attempt restricted tool call → must be blocked

### Sprint 2: Outbound Information Filter (R-01)
**Weighted Score: 4.6**
**Remediation: `06-outbound-info-filter.md` + `00-information-disclosure.md`**
**Status: ❌ No module exists**

- [ ] New module: `OutboundInfoFilter` in gateway response pipeline
- [ ] Classify outbound content: system info, PII, operational data, safe
- [ ] Redact before delivery: hostnames, tool names, infrastructure topology, credential paths, user IDs, internal URLs
- [ ] Allowlist of safe-to-disclose categories
- [ ] System prompt restriction (soft control, complement to hard enforcement)
- [ ] Block `function_calls` XML from leaking in responses

**Blocked categories:**
- Infrastructure: hostnames, IPs, Tailscale domains, ports, URLs
- Security: module names, modes, thresholds, config details
- Tools: MCP tool names, parameter formats
- Users: Telegram IDs, authorized user lists
- Operations: active bugs, version numbers, credential paths

**Verification:**
- Ask "what tools do you have?" → general description, no MCP tool names
- Ask about infrastructure → no hostnames or IPs
- Ask about security → no module names or modes

---

### Sprint 3: Human-in-the-Loop (R-08, R-09)
**Weighted Score: 4.4**
**Remediation: `02-human-in-the-loop.md`**
**Status: ❌ Approval Queue exists but not wired**

- [ ] Define MCP tool risk tiers in gateway config:
  - **Critical:** `exec`, `cron`, `sessions_send`
  - **High:** `nodes`, `browser`, SSH commands
  - **Medium:** file writes, iCloud access, email send
  - **Low:** read operations, search, status checks
- [ ] Wire MCP Proxy (#19) to route high/critical tools to Approval Queue (#3)
- [ ] Approval Queue in enforce mode: hold in SQLite, push via WebSocket to dashboard
- [ ] Auto-deny after configurable timeout (default 5 min)
- [ ] Telegram inline buttons for approve/deny (owner only)
- [ ] Bypass for owner-initiated requests (optional, configurable)

**Verification:**
- Agent attempts `exec` → queued for approval, not executed
- Agent attempts email send → queued
- Timeout → auto-denied
- Owner approves → executed

---

### Sprint 4: Per-User Session Isolation (R-04, R-05)
**Weighted Score: 4.2**
**Remediation: `03-session-isolation.md`**
**Status: ❌ Single-tenant, all users share context**

- [ ] Session manager maps Telegram user ID to isolated context
- [ ] Per-user: conversation history, file workspace, persistent memory
- [ ] Per-user trust level tracking (Progressive Trust #9 scoped to user)
- [ ] Data from User A inaccessible in User B's session
- [ ] Cross-user data sharing requires explicit consent
- [ ] Owner can view all sessions (admin override)

**Verification:**
- User A shares data → User B cannot access it
- User A's files not in User B's workspace
- Trust levels independent per user

---

### Sprint 5: Separation of Privilege (R-06, R-07)
**Weighted Score: 4.2**
**Remediation: `04-separation-of-privilege.md`**
**Status: ⚠️ Read-only rootfs helps, but SSH access bypasses**

- [ ] Mount gateway source, config, security policies as `:ro` Docker volumes
- [ ] Add AgentShroud source paths to FileSandbox (#23) deny list (enforce mode) ✅ (partially done)
- [ ] Block SSH commands targeting the proxy host
- [ ] Security file modifications require out-of-band human authorization
- [ ] Agent cannot modify: `SOUL.md`, system prompts, module configs, gateway source

**Verification:**
- Agent attempts to write gateway config → blocked
- Agent attempts SSH to gateway host → blocked
- Agent attempts to modify SOUL.md → blocked

---

### Sprint 6: Credential Isolation (R-10, R-11, R-12)
**Weighted Score: 4.0**
**Remediation: `05-credential-isolation.md`**
**Status: ⚠️ Op-proxy exists but secrets still mounted in agent container**

- [ ] Remove ALL secret mounts from agent (OpenClaw) container
- [ ] Move all credentials to gateway Docker Secrets exclusively
- [ ] Transparent credential injection: gateway matches outbound requests by destination, injects auth headers
- [ ] Agent requests go through gateway proxy → credentials never visible to agent
- [ ] Audit all current secret mounts (`/run/secrets/`, env vars)
- [ ] Op-proxy becomes the ONLY credential access path

**Verification:**
- Agent runs `ls /run/secrets/` → empty or nonexistent
- Agent runs `env | grep KEY` → no credentials
- API calls still work (gateway injects credentials transparently)

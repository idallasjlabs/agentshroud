# v0.8.0 Feature: Interactive Egress Firewall ("Little Snitch for Agents")

## Concept
All outbound network traffic requires explicit human approval. Like Little Snitch on macOS, but for AI agent connections.

## User Experience
When an agent tries to reach a new domain:
1. **Prompt owner** via Telegram/chat with:
   - Domain name (e.g., `api.stripe.com:443`)
   - Service/tool requesting access (e.g., `web_fetch`, `http_request`)
   - Agent ID making the request
   - Risk assessment: 🟢 Green (known safe), 🟡 Yellow (unknown), 🔴 Red (suspicious)
2. **Owner chooses**:
   - ✅ Allow permanently (add to allowlist)
   - ⏱️ Allow temporarily (until session end / restart / N minutes)
   - 🚫 Deny permanently (add to denylist)
   - ❌ Deny once

## Console Management
- View all allowed/denied domains with approval type and timestamp
- Revoke permanent approvals
- See recent connection attempts (allowed + denied)
- Per-agent allowlist views
- Bulk approve/deny

## Architecture (high-level)
- Builds on existing `EgressFilter` + `EgressFilterConfig` from v0.7.0
- New: `EgressApprovalQueue` — holds pending requests, notifies owner
- New: `/manage/egress/rules` — CRUD for firewall rules
- New: `/manage/egress/pending` — pending approval requests
- New: Telegram inline buttons for approve/deny
- Persist rules to `egress_rules.json` (survives restart)
- Temporary rules tracked in memory with TTL

## Risk Assessment Heuristic
- 🟢 Green: domain in well-known safe list (api.openai.com, api.anthropic.com, github.com, icloud.com)
- 🟡 Yellow: unknown domain, standard ports (80/443)
- 🔴 Red: IP address (not domain), non-standard ports, known malicious TLDs, matches exfil patterns

## Feature: Global Monitor-Only Mode ("Observatory Mode")

### Concept
A single switch that puts ALL 33 security modules into monitor-only mode simultaneously. Every module logs threats and anomalies but never blocks. This creates a transparent observation layer over vanilla OpenClaw — perfect for blue/red team testing.

### Use Cases
1. **Baseline Testing**: Run OpenClaw with AgentShroud observing but not interfering. Measure what vanilla behavior looks like under attack.
2. **Red Team Assessment**: Steve Hay's STPA-Sec probes run unblocked while all 33 modules log what they *would* have blocked. Generates a complete threat matrix.
3. **Blue Team Comparison**: Run the same workload in monitor vs enforce mode. Diff the logs to prove each module's value-add.
4. **Regression Testing**: Verify modules detect threats correctly without worrying about false-positive blocking.
5. **Onboarding**: New deployments start in monitor mode to build allowlists before switching to enforce.

### Design

```
# Environment variable or config
AGENTSHROUD_GLOBAL_MODE=monitor|enforce   # Default: enforce

# API endpoint
POST /manage/mode {"mode": "monitor"}     # Switch all modules
GET  /manage/mode                         # Current mode + per-module overrides
```

### Key Behaviors
- **Global override**: Sets all 33 modules to `monitor` regardless of individual config
- **Per-module override**: Individual modules can still be pinned to `enforce` even in global monitor mode (e.g., keep killswitch active)
- **Audit trail**: Mode changes logged as security events with timestamp, who changed it, reason
- **Visual indicator**: Web dashboard shows prominent banner: "⚠️ MONITOR MODE — No threats blocked"
- **TUI indicator**: Status bar turns yellow with "OBSERVING" label
- **Auto-revert**: Optional timer to auto-switch back to enforce after N hours (safety net)
- **Log format**: Every would-block event tagged `[MONITOR]` with full details of what enforcement would have done

### API Response
```json
{
  "global_mode": "monitor",
  "effective_since": "2026-02-25T15:45:00Z",
  "auto_revert_at": "2026-02-26T15:45:00Z",
  "modules": {
    "pii_sanitizer": {"configured": "enforce", "effective": "monitor"},
    "file_sandbox": {"configured": "enforce", "effective": "monitor"},
    "killswitch": {"configured": "enforce", "effective": "enforce", "pinned": true},
    ...
  }
}
```

### Implementation Priority
**P1** — This should ship alongside or before the egress firewall. It's foundational for all testing workflows and relatively simple to implement (each module already has a mode field).

### Estimated Effort
- SecurityPipeline: Add global mode override (~2h)
- Each module: Respect global mode in check methods (~4h total for 33)
- API endpoints: mode get/set/auto-revert (~2h)  
- Dashboard/TUI: Mode indicator + toggle (~3h)
- Tests: Mode switching, per-module override, auto-revert (~3h)
- **Total: ~14h**

## 2) Additional v0.8.0 Stabilization Work (Current Execution Cycle)

### Telegram security/UX hardening delivered
- Canonical protected header normalized to:
  - `🛡️ Protected by AgentShroud` + two newlines.
- Collaborator-safe response templates tightened to:
  - prevent command/file/secret/tool output exposure,
  - keep conceptual guidance available,
  - consistently indicate owner-gated capability boundaries.
- Owner/collaborator local deterministic command handling reinforced:
  - `/start`, `/help`, `/status`, `/whoami`, `/model`, healthcheck aliases.
- Owner admin command ergonomics strengthened:
  - target resolution by numeric ID, static alias, and pending username alias (`/approve ana`, `/deny ana`).
- Unknown/revoked user onboarding flow hardened:
  - deterministic pending-approval notices and owner notification flow.
- Outbound anti-leak filtering strengthened:
  - suppression/redaction of raw tool JSON/XML/internal approval banners for collaborator channels.
- Fallback reliability paths added:
  - blocked notice and local notice send retries with deterministic backup text (no silent drop class).

### Assessment/quality loop support delivered
- Expanded regression coverage in gateway inbound/outbound test suites for:
  - alias-based owner approvals/denials,
  - local command normalization variants,
  - collaborator pending notice handling,
  - fallback behavior on send failures,
  - egress redaction wording requirements.

## 6) Owner Readout — Daily Telegram Validation Checklist

Use this as a fast operational check after deploy/restart.

### A. Startup + baseline
1. Owner receives:
   - `🟡 AgentShroud starting`
   - `🛡️ AgentShroud online`
2. Owner `/status` returns deterministic status response.
3. Owner `/whoami` returns role + user id.

### B. Collaborator onboarding
1. Unknown user `/start` gets:
   - `🛡️ Protected by AgentShroud`
   - pending owner approval message.
2. Owner gets pending request notice.
3. Owner approves with `/approve <user_id|name>`.
4. Collaborator `/start` now returns collaborator-ready protected message.

### C. Command contract (role split)
1. Owner commands work locally:
   - `/help`, `/status`, `/whoami`, `/pending`, `/collabs`.
2. Collaborator allowed commands respond:
   - `/help`, `/status`, `/whoami`, `/model`.
3. Collaborator blocked command (e.g., `/exec whoami`) returns protected denial (no tool output).

### D. Egress behavior
1. Collaborator asks for external URL:
   - collaborator gets protected owner-gated response.
2. Owner receives egress approval request (no collaborator visibility of internal approval details).

### E. Leak-safety checks
1. No raw JSON/tool payload in user-visible chat:
   - no `{"name":...,"arguments":...}`
   - no `<function_calls>...</function_calls>`
2. No flash-then-redact behavior observed.

### F. Rate-limit UX
1. If collaborator is limited, response explicitly states rate limit + retry guidance.
2. No silent no-response behavior.

### G. Daily evidence capture
- Run and archive:
  - `python security_assessment/run_assessment.py`
- Confirm reports are timestamped and stored in the report directory.
- Record any regressions and link to failing prompt/response pair.

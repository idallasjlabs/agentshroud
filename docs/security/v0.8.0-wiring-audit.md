> **Method:** Source code trace of every security module from class definition → instantiation → request path

## Executive Summary

**92 features committed. 90 complete. 2 in progress. But several "complete" features have wiring gaps where the module exists, tests pass, but the module is NOT connected to the live request flow.**

The pattern is consistent: classes are well-implemented and well-tested, but the glue code in `lifespan.py` that connects them to the runtime was never written for some modules.

## 🔴 CRITICAL — Not Wired (code exists, tests pass, but NOT running in production)

### 1. EgressTelegramNotifier — Little Snitch Inline Buttons
- **File:** `gateway/proxy/telegram_egress_notify.py`
- **Class:** `EgressTelegramNotifier`
- **Problem:** Class is NEVER instantiated. Not imported in `lifespan.py`, `pipeline.py`, or anywhere in runtime. The egress filter silently blocks/allows without ever notifying the owner via Telegram.
- **Impact:** The entire "Little Snitch for Agents" UX (features D.38-44) doesn't work as designed. Egress filtering works but there are NO inline approve/deny buttons, NO Telegram notifications for blocked domains, and NO way to approve unknown domains interactively.
- **What exists:**
  - `EgressTelegramNotifier` class with `notify_pending()`, `handle_callback()`, `answer_callback()`
  - Inline keyboard buttons (Allow Always / Allow Once / Deny)
  - Callback data parsing (egress_allow_always_, egress_allow_once_, egress_deny_)
  - Tests pass
- **What's missing:**
  - Instantiate `EgressTelegramNotifier` in `lifespan.py` with bot_token + owner_chat_id
  - Connect to `EgressFilter.check()` — unknown domains should trigger notification instead of silent block
  - Register callback handler in `telegram_proxy.py` for `callback_query` updates with `egress_*` data
  - Async wait mechanism: `EgressFilter.check()` should await owner decision (up to 30s timeout)
  - Revoke/manage mechanism for approved domains
- **Fix complexity:** Medium-high (2-4 hours)

### 2. Approval Queue — Missing Telegram Notifications
- **File:** `gateway/approval_queue/enhanced_queue.py`
- **Class:** `EnhancedApprovalQueue`
- **Problem:** Queue is instantiated WITHOUT `bot_token` or `admin_chat_id` parameters. The `_notify_telegram()` method exists and works, but silently skips because credentials are never provided.
- **Impact:** High-risk tool calls go into the approval queue but the owner is NEVER notified via Telegram. They'd have to check the dashboard or API manually to see pending approvals.
- **What exists:**
  - `_notify_telegram()` method with httpx POST to Telegram API
  - `bot_token` and `admin_chat_id` constructor params
  - Notification skips gracefully when not configured
- **What's missing:**
  - Pass `bot_token` and `admin_chat_id` to `EnhancedApprovalQueue()` in `lifespan.py`
  - These should come from env vars (`TELEGRAM_BOT_TOKEN`, owner chat ID from `RBACConfig`)
- **Fix complexity:** Low (15 minutes — just pass 2 params)

### 3. OutputCanary — Not Passed to Pipeline
- **File:** `gateway/security/output_canary.py`
- **Class:** `OutputCanary`
- **Problem:** Instantiated in `middleware.py` (line 338) with getter `get_output_canary()`, but the getter is NEVER called. The `SecurityPipeline` accepts `output_canary=` param but it's not passed during construction in `lifespan.py`.
- **Impact:** The pipeline's Step 1.8 (canary leak detection in outbound responses) never fires. Canary tokens may be planted but leaks in responses go undetected.
- **Note:** `CanaryTripwire` IS wired (different module — checks inbound). OutputCanary checks outbound. Both are needed.
- **What's missing:**
  - In `lifespan.py`, add `output_canary=app_state.middleware_manager.get_output_canary()` to `SecurityPipeline()` constructor
- **Fix complexity:** Low (5 minutes — one line)

### 4. EnhancedToolResultSanitizer — Not Passed to Pipeline
- **File:** `gateway/security/tool_result_sanitizer_enhanced.py`
- **Class:** `ToolResultSanitizer` (aliased as `EnhancedToolResultSanitizer`)
- **Problem:** Instantiated in `middleware.py` (line 362) with getter `get_enhanced_tool_sanitizer()`, but getter is NEVER called. Pipeline accepts `enhanced_tool_sanitizer=` param but it's not passed.
- **Impact:** Pipeline's Step 1.75 (enhanced markdown-aware PII sanitization on outbound content) never fires. The basic `ToolResultSanitizer` in middleware may still run on tool results, but the enhanced version that handles markdown tables, code blocks, and link injection doesn't run in the pipeline's outbound path.
- **What's missing:**
  - In `lifespan.py`, add `enhanced_tool_sanitizer=app_state.middleware_manager.get_enhanced_tool_sanitizer()` to `SecurityPipeline()` constructor
- **Fix complexity:** Low (5 minutes — one line)

### 5. LLMProxy — Never Instantiated
- **File:** `gateway/proxy/llm_proxy.py`
- **Class:** `LLMProxy`
- **Problem:** Referenced in `main.py` (lines 2335-2370) for `/llm/proxy` and `/llm/proxy/stats` endpoints, but NEVER instantiated in `lifespan.py`. All LLM proxy requests return "LLM proxy not initialized".
- **Impact:** The LLM request proxying route (which would apply middleware security to LLM API calls) is a dead endpoint.
- **Fix complexity:** Medium (1 hour — needs config for target LLM endpoints)

## 🟡 MEDIUM — Partially Wired or Degraded

### 6. Callback Query Handler for Inline Buttons
- **Problem:** `telegram_proxy.py` processes Telegram updates but does NOT handle `callback_query` updates. When inline buttons are pressed (egress approve/deny, approval queue decisions), there's no handler to process them.
- **Impact:** Even if we wire up EgressTelegramNotifier, button presses would go unprocessed.
- **What's missing:** Add callback_query processing in `telegram_proxy.py`'s `_filter_inbound_updates()` or a dedicated route in `main.py`
- **Fix complexity:** Medium (1-2 hours)

### 7. KeyVault — Instantiated but Not Used
- **File:** `gateway/security/key_vault.py`
- **Problem:** `KeyVault` is instantiated in `lifespan.py` (line 425) and stored on `app_state`, but no other runtime code reads from or writes to it. It initializes successfully but sits idle.
- **Impact:** Low — credential injection works via `CredentialInjector` and 1Password. KeyVault is a secondary in-memory store that's ready but unused.
- **Fix complexity:** Low (wire into CredentialInjector as a cache layer)

### 8. SidecarScanner — External Tool Interface
- **File:** `gateway/proxy/sidecar.py`
- **Problem:** Not instantiated. Designed for external security scanning sidecars (ClamAV, etc.) via HTTP API.
- **Impact:** Low — ClamAV, Trivy, etc. are called directly via their native CLIs, not through this abstraction.
- **Fix complexity:** Low but unnecessary — CLI integration works fine

### 9. Audit Store Not Passed to Pipeline
- **Problem:** `SecurityPipeline` accepts `audit_store=` param but it's not passed in `lifespan.py`. The pipeline uses its internal `AuditChain` which now persists to SQLite, but the external `AuditStore` (used by other modules) isn't connected.
- **Impact:** Medium — audit chain works internally but isn't integrated with the centralized `AuditStore` used by the alert dispatcher and health report.
- **Fix complexity:** Low (pass audit_store to pipeline constructor)

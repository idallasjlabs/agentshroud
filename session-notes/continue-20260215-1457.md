## Key Decisions & Context

### 1. API Key Loading via Startup Script
**Decision**: Use a Bash wrapper script instead of entrypoint overrides in docker-compose.yml.

**Rationale**: Docker Compose entrypoint/command syntax kept breaking. The startup script approach is cleaner, more maintainable, and properly handles the Docker `ENTRYPOINT` + `CMD` pattern. Script is baked into the Dockerfile so it survives rebuilds.

### 2. Telegram dmPolicy: "open" (Temporary)
**Decision**: Set dmPolicy to "open" with `allowFrom: ["*"]` instead of fighting with pairing codes.

**Rationale**: Pairing codes were expiring before approval could be executed. This is a **temporary workaround** for MVP testing. Plan Phase 3A.8 explicitly addresses changing this to "allowlist" mode with the user's Telegram ID (8096968754).

**Risk**: Bot is publicly accessible via Telegram. Acceptable for testing, not for production.

### 3. Deferred Egress Filtering to Phase 4
**Decision**: Document the lack of outbound network filtering but don't implement it yet.

**Rationale**: On macOS with Docker Desktop, container traffic flows through a Linux VM, making host-level packet filtering unreliable. A proper solution requires a Squid/nginx proxy container (Phase 7). Documented as accepted risk for MVP.

### 4. Plan Structure: 8 Phases
**Decision**: Break implementation into clear phases with dependencies.

**Rationale**: User lost track of Phase 3 progress ("lost the plot"). The phased approach ensures we complete critical security gaps (3A) before adding new features. Each phase has concrete deliverables and verification steps.

### 5. Two Tailscale Options
**Decision**: Document both `tailscale serve` (recommended) and port binding options.

**Rationale**: User asked for access via `http://marvin.tail240ea8.ts.net:18790`. Tailscale serve is more secure (keeps localhost binding), but binding to Tailscale IP is simpler. Documented both.

### 6. Reuse Existing Gateway Test Patterns
**Decision**: All new code (SSH proxy, kill switch, dashboard) should follow the existing test patterns in `gateway/tests/`.

**Rationale**: Gateway already has 89% coverage with 87 passing tests. The existing patterns (pytest fixtures, async tests, mock docker commands) are proven and well-documented.
